# 수동 테스트 시나리오

## 공통 테스트 데이터 가정
- 활성 사용자 2명
  - 사용자 A: `user-a@company.com`
  - 사용자 B: `user-b@company.com`
- 활성 회의실 2개 이상
  - 회의실 1: Room A
  - 회의실 2: Room B
- 사용자 A와 사용자 B는 모두 로그인 가능한 비밀번호를 알고 있다.
- 아래 예약 데이터가 사전에 준비되어 있다.
  - 사용자 A의 미래 예약 1건: 오늘 이후 날짜, Room A, 10:00~11:00
  - 사용자 A의 과거 시작 예약 1건: 오늘 날짜, 현재 시각보다 이전에 시작한 예약
  - 사용자 B의 미래 예약 1건: 오늘 이후 날짜, Room A, 13:00~14:00

## TC-01 로그인 성공

### 테스트 항목명
- 로그인 성공

### 사전 조건
- 사전 등록된 활성 사용자 계정이 존재한다.
- 로그인 화면에 접근 가능한 상태다.

### 테스트 절차
1. 로그인 화면으로 이동한다.
2. 회사 이메일에 유효한 사용자 계정을 입력한다.
3. 비밀번호에 해당 계정의 올바른 비밀번호를 입력한다.
4. 로그인 버튼을 클릭한다.

### 기대 결과
- 로그인에 성공한다.
- 메인 예약 현황 화면으로 이동한다.
- 메인 타임라인 화면에서 날짜 선택, 회의실별 타임라인, 예약 블록이 표시된다.

## TC-02 로그인 실패

### 테스트 항목명
- 로그인 실패 메시지 노출

### 사전 조건
- 로그인 화면에 접근 가능한 상태다.

### 테스트 절차
1. 로그인 화면으로 이동한다.
2. 회사 이메일에 존재하는 계정 또는 임의 이메일을 입력한다.
3. 비밀번호에 잘못된 값을 입력한다.
4. 로그인 버튼을 클릭한다.

### 기대 결과
- 로그인되지 않는다.
- 로그인 실패 메시지가 표시된다.
- 현재 화면에 그대로 머문다.

## TC-03 비밀번호 변경 성공

### 테스트 항목명
- 내 정보 화면에서 비밀번호 변경 성공

### 사전 조건
- 사용자 A로 로그인된 상태다.
- 현재 비밀번호를 알고 있다.

### 테스트 절차
1. 내 정보 화면으로 이동한다.
2. 비밀번호 변경 영역을 찾는다.
3. 현재 비밀번호를 입력한다.
4. 새 비밀번호를 입력한다.
5. 변경 버튼을 클릭한다.
6. 로그아웃한다.
7. 새 비밀번호로 다시 로그인한다.

### 기대 결과
- 비밀번호 변경 성공 메시지가 표시된다.
- 로그아웃 후 새 비밀번호로 정상 로그인된다.
- 첫 로그인 강제 변경 안내나 별도 유도 팝업은 표시되지 않는다.

3. 새 비밀번호를 입력한다.
4. 변경 버튼을 클릭한다.

### 기대 결과

## TC-05 날짜 이동

### 테스트 항목명
- 메인 타임라인 날짜 이동

### 사전 조건
- 로그인된 상태다.
- 메인 예약 현황 화면에 접근 가능한 상태다.

### 테스트 절차
1. 메인 화면에서 현재 선택된 날짜를 확인한다.
2. `다음 날` 버튼을 클릭한다.
3. 날짜 입력값이 변경되는지 확인한다.
4. `이전 날` 버튼을 클릭한다.
5. 날짜 입력 필드에 특정 날짜를 직접 선택한다.

### 기대 결과
- 날짜 이동 버튼 클릭 시 선택 날짜가 변경된다.
- 날짜 직접 선택 시 해당 날짜 기준 타임라인으로 갱신된다.
- 날짜 이동이 사용자가 쉽게 인지 가능한 방식으로 동작한다.

## TC-06 빈 슬롯 클릭 예약 생성 성공

### 테스트 항목명
- 빈 슬롯 클릭 기반 예약 생성

### 사전 조건
- 사용자 A로 로그인된 상태다.
- 미래 날짜의 Room B에 빈 슬롯이 존재한다.

### 테스트 절차
1. 메인 예약 현황 화면에서 미래 날짜를 선택한다.
2. Room B의 비어 있는 슬롯 하나를 클릭한다.
3. 예약 생성 모달이 열리는지 확인한다.
4. 모달에 회의실, 날짜, 시작 시간이 자동 반영되는지 확인한다.
5. 종료 시간을 시작 시간보다 뒤의 값으로 선택한다.
6. 예약 목적을 선택적으로 입력한다.
7. 저장 버튼을 클릭한다.

### 기대 결과
- 예약 생성 모달이 열린다.
- 회의실, 날짜, 시작 시간이 기본값으로 채워진다.
- 예약 생성 성공 메시지가 표시된다.
- 생성된 예약이 메인 타임라인에 즉시 반영된다.

## TC-07 이미 예약된 슬롯 클릭 처리

### 테스트 항목명
- 이미 예약된 슬롯은 빈 슬롯으로 선택되지 않음

### 사전 조건
- 사용자 B의 미래 예약이 Room A 13:00~14:00에 존재한다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 해당 예약이 있는 날짜로 이동한다.
2. Room A의 13:00~14:00 구간을 확인한다.
3. 예약 블록이 아닌 빈 슬롯 셀 클릭 방식으로 해당 구간을 선택해 본다.
4. 예약 블록 자체를 클릭한다.

### 기대 결과
- 예약된 슬롯 영역은 빈 슬롯으로 클릭되지 않는다.
- 예약 생성 모달은 열리지 않는다.
- 예약 블록 클릭 시 상세 모달이 열린다.

## TC-08 남의 예약 상세 조회

### 테스트 항목명
- 남의 예약은 조회 전용 상세만 가능

### 사전 조건
- 사용자 B의 미래 예약이 존재한다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 메인 화면에서 사용자 B의 예약 블록을 클릭한다.
2. 상세 모달의 표시 정보를 확인한다.
3. 편집 버튼과 취소 버튼 노출 여부를 확인한다.

### 기대 결과
- 상세 모달이 열린다.
- 회의실명, 예약자 이름, 날짜, 시작 시간, 종료 시간, 예약 목적이 표시된다.
- 예약자 이메일은 표시되지 않는다.
- 사용 시간은 표시되지 않는다.
- 편집 버튼과 취소 버튼은 표시되지 않는다.

### 테스트 항목명

### 기대 결과

## TC-10 내 예약 편집 성공

### 테스트 항목명
- 시작 전 내 예약 편집 성공

### 사전 조건
- 사용자 A의 미래 예약이 존재한다.
- 예약 시작 시간이 아직 지나지 않았다.

### 테스트 절차
1. 내 예약 상세 모달을 연다.
2. 편집 버튼을 클릭한다.
3. 동일 모달 안에서 편집 모드로 전환되는지 확인한다.
4. 시작 시간 또는 종료 시간을 다른 유효한 값으로 변경한다.
5. 예약 목적을 변경한다.
6. 저장 버튼을 클릭한다.

### 기대 결과
- 별도 새 모달이 아니라 같은 모달에서 편집 모드로 전환된다.
- 회의실은 변경할 수 없다.
- 수정 성공 메시지가 표시된다.
- 수정된 시간이 타임라인에 즉시 반영된다.

## TC-11 시작 시간 지난 예약 편집 제한

### 테스트 항목명
- 시작 시간이 지난 예약은 수정 불가

### 사전 조건
- 사용자 A의 예약 중 시작 시간이 이미 지난 예약이 존재한다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 내 예약 상세 모달을 연다.
2. 편집 버튼 상태를 확인한다.
3. 편집을 시도한다.

### 기대 결과
- 예약 시작 시간이 지난 뒤에는 수정할 수 없다는 안내가 표시된다.
- 수정 기능은 사용할 수 없다.

## TC-12 예약 취소

### 테스트 항목명
- 내 예약 취소 성공

### 사전 조건
- 사용자 A의 예약이 존재한다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 내 예약 상세 모달을 연다.
2. 취소 버튼을 클릭한다.
3. 확인 모달이 한 번 더 표시되는지 확인한다.
4. 취소 확정 버튼을 클릭한다.

### 기대 결과
- 취소 확인 모달이 추가로 표시된다.
- 취소 성공 후 예약 블록이 타임라인에서 즉시 사라진다.
- 취소는 시작 시간과 무관하게 수행 가능하다.

## TC-13 중복 예약 방지

### 테스트 항목명
- 동일 회의실 중복 예약 차단

### 사전 조건
- Room A에 기존 예약이 10:00~11:00으로 존재한다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 같은 날짜, 같은 회의실에서 10:30 시작 빈 슬롯 또는 편집 가능한 예약을 기준으로 예약/수정을 시도한다.
2. 종료 시간을 11:30으로 선택한다.
3. 저장을 시도한다.
4. 다시 11:00 시작, 12:00 종료 조건으로 예약/수정을 시도한다.

### 기대 결과
- 10:30~11:30은 중복으로 판단되어 생성 또는 수정되지 않는다.
- 즉시 안내 또는 저장 시 실패 메시지로 예약 불가가 전달된다.
- 11:00~12:00은 허용된다.

## TC-14 다른 회의실 동시 예약 허용

### 테스트 항목명
- 다른 회의실 동시간대 예약 허용

### 사전 조건
- Room A에 10:00~11:00 예약이 존재한다.
- 같은 날짜의 Room B는 동일 시간대가 비어 있다.
- 사용자 A로 로그인된 상태다.

### 테스트 절차
1. 같은 날짜의 Room B에서 10:00 시작 빈 슬롯을 클릭한다.
2. 종료 시간을 11:00으로 선택한다.
3. 저장한다.

### 기대 결과
- 다른 회의실이므로 예약 생성이 허용된다.
- 타임라인에 Room B 예약이 즉시 표시된다.

## TC-15 반응형 UI - 태블릿

### 테스트 항목명
- 태블릿 해상도에서 타임라인과 모달 사용 가능

### 사전 조건
- 로그인된 상태다.
- 브라우저 개발자 도구 또는 실제 태블릿 기기에서 확인 가능하다.

### 테스트 절차
1. 화면 폭을 태블릿 수준으로 조정한다.
2. 메인 타임라인 화면의 날짜 선택, 회의실 목록, 시간축을 확인한다.
3. 가로/세로 스크롤 사용성을 확인한다.
4. 빈 슬롯 클릭 후 생성 모달을 열어 입력 가능한지 확인한다.
5. 예약 블록 클릭 후 상세 모달을 확인한다.

### 기대 결과
- 기본 구조가 유지된다.
- 스크롤 UX가 과도하게 불편하지 않다.
- 생성/조회/수정/취소 모달 사용이 가능하다.

## TC-16 반응형 UI - 모바일

### 테스트 항목명
- 모바일 해상도에서 타임라인과 모달 사용 가능

### 사전 조건
- 로그인된 상태다.
- 브라우저 개발자 도구 또는 실제 모바일 기기에서 확인 가능하다.

### 테스트 절차
1. 화면 폭을 모바일 수준으로 조정한다.
2. 메인 타임라인 화면에서 날짜 선택이 가능한지 확인한다.
3. 회의실 확인이 가능한지 확인한다.
4. 필요한 경우 가로 스크롤로 시간대를 탐색한다.
5. 빈 슬롯을 클릭해 생성 모달을 열고 종료 시간/예약 목적 입력을 시도한다.
6. 예약 블록을 클릭해 상세 모달을 확인한다.

### 기대 결과
- 타임라인이 필요한 경우 가로 스크롤로 탐색 가능하다.
- 날짜 선택과 회의실 확인이 가능하다.
- 슬롯 클릭 후 예약 입력이 무리 없이 가능하다.
- 정보 과밀로 인해 조회/예약이 어려운 수준이 아니어야 한다.

### 테스트 절차
1. 내 정보 화면으로 이동한다.
2. 사용자 기본 정보 표시 여부를 확인한다.
3. 내 예약 목록이 표시되는지 확인한다.
4. 예약 목록 컬럼을 확인한다.
5. 예약 행을 클릭해 상세 모달이 열리는지 확인한다.
6. 비밀번호 변경 기능이 같은 화면에 존재하는지 확인한다.

### 기대 결과
- 사용자 기본 정보가 표시된다.
- 내 예약 목록이 표시된다.
- 목록에는 회의실명, 날짜, 시작 시간, 종료 시간, 예약 목적, 예약 상태가 표시된다.
- 예약 상세 확인 흐름으로 이어질 수 있다.
- 비밀번호 변경 기능이 제공된다.
