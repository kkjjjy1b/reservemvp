프로젝트명
사내 회의실 예약 웹 서비스 MVP

프로젝트 목표
회사 구성원이 웹에서 회의실 예약 현황을 직관적으로 확인하고,
비어 있는 시간 슬롯을 클릭하여 중복 없이 회의실을 예약할 수 있는 반응형 웹 서비스를 만든다.

핵심 전제
- 이 서비스는 사내 구성원 전용 서비스이다.
- 사용자 식별자는 회사 이메일이다.
- 회사 이메일은 회원 DB 내 unique 값이다.
- 회원은 관리자에 의해 사전에 DB에 등록된다.
- 최초 로그인은 회사 이메일 + 공통 초기 비밀번호 방식으로 진행한다.
- 사용자는 로그인 후 비밀번호를 변경할 수 있어야 한다.
- 초기 비밀번호 변경은 권장하지 않으며 별도 안내 기능도 구현하지 않는다.

핵심 사용자
- 회사 내 회의실을 사용하는 일반 구성원

핵심 사용자 시나리오
1. 사용자는 회사 이메일과 비밀번호로 로그인한다.
2. 사용자는 메인 화면에서 특정 날짜의 회의실 예약 현황을 확인한다.
3. 화면은 회의실별 타임라인 형태로 구성되며 30분 단위 슬롯이 표시된다.
4. 사용자는 비어 있는 슬롯을 클릭한다.
5. 클릭한 슬롯 정보를 기반으로 예약 생성 모달이 열린다.
6. 모달에는 선택된 날짜, 시작 시간, 회의실 정보가 기본 반영된다.
7. 사용자는 종료 시간과 예약 목적을 입력한다.
8. 시스템은 동일 회의실의 시간 중복 여부를 서버에서 최종 검증한다.
9. 중복이 없으면 예약이 생성된다.
10. 생성된 예약은 즉시 타임라인에 반영된다.
11. 이미 예약된 슬롯은 클릭할 수 없다.
12. 사용자는 예약 블록을 클릭하여 예약 상세 내용을 확인할 수 있다.
13. 내 예약인 경우에만 편집 또는 취소가 가능하다.
14. 남의 예약은 클릭 가능하지만 조회만 가능하다.
15. 사용자는 내 정보 화면에서 내 예약 목록과 비밀번호 변경 기능을 사용할 수 있다.

핵심 기능 요구사항

[인증/회원]
- 사용자는 회사 이메일과 비밀번호로 로그인할 수 있어야 한다.
- 회사 이메일은 회원의 고유 식별자이며 unique 제약을 가져야 한다.
- 회원은 회원가입 방식이 아니라 관리자가 사전에 DB에 insert 하는 방식으로 생성한다.
- 모든 회원은 최초에 동일한 공통 초기 비밀번호를 가진다.
- 비밀번호는 해시 형태로 저장해야 한다.
- 사용자는 로그인 후 비밀번호를 변경할 수 있어야 한다.
- 비밀번호 변경 안내 배너, 첫 로그인 유도 팝업 등은 구현하지 않는다.

[회의실]
- 시스템은 여러 개의 회의실을 관리할 수 있어야 한다.
- 회의실 수는 최대 4개 수준을 가정한다.
- 각 회의실은 최소한 아래 정보를 가진다:
  - 이름
  - 위치(선택)
  - 설명(선택)
- 사용자는 날짜별 타임라인에서 회의실 단위로 예약 가능 여부를 확인할 수 있어야 한다.

[예약]
- 사용자는 빈 시간 슬롯을 클릭하여 예약을 생성할 수 있어야 한다.
- 예약 생성 시 필요한 정보:
  - 회의실
  - 예약 날짜
  - 시작 시간
  - 종료 시간
  - 예약 목적(선택 입력)
- 예약 시작 시각은 타임라인 슬롯 클릭으로 결정된다.
- 사용자는 예약 종료 시각을 선택해야 한다.
- 시작 시간과 종료 시간은 30분 단위여야 한다.
- 최소 예약 단위는 30분이다.
- 최대 예약 시간 제한은 없다.
- 동일 회의실에서 시간이 겹치는 예약은 생성될 수 없어야 한다.
- 다른 회의실은 같은 시간대에 동시에 예약 가능하다.
- 같은 사용자가 같은 시간대에 다른 회의실을 중복 예약하는 것도 허용한다.
- 예약이 성공하면 즉시 타임라인에 반영되어야 한다.

[예약 시간 정책]
- 타임라인에서 표시하는 시간 범위는 매일 06:00부터 24:00까지이다.
- 과거 날짜는 예약 생성 불가이다.
- 오늘 날짜의 현재 시각 이전 시작 시간은 예약 생성 불가이다.
- 기존 예약 수정은 예약 시작 전까지만 가능하다.
- 예약 시작 시간이 지난 뒤에는 수정할 수 없다.
- 예약 취소는 시작 시간과 무관하게 언제든 가능하다.

[예약 수정]
- 내 예약일 때만 수정 가능하다.
- 수정 가능한 항목:
  - 시작 시간
  - 종료 시간
  - 예약 목적
- 회의실 변경은 불가하다.
- 예약 목적은 선택 입력이다.
- 수정 시에도 시작 시간과 종료 시간은 30분 단위여야 한다.
- 수정 시에도 서버에서 동일 회의실의 중복 예약 여부를 최종 검증해야 한다.
- 수정하려는 예약이 이미 수정 가능 시간을 지났다면 정확한 안내 메시지를 보여줘야 한다.
- 수정 UI는 별도 모달이 아니라 같은 상세 모달 안에서 편집 모드로 전환되는 방식이다.

[예약 취소]
- 내 예약일 때만 취소 가능하다.
- 취소 버튼 클릭 시 확인 모달을 한 번 더 보여줘야 한다.
- 취소 확정 시 예약은 즉시 취소된다.
- 취소된 예약은 타임라인에서 즉시 사라져야 한다.

[예약 충돌 규칙]
- 같은 회의실에서 예약 시간 구간이 조금이라도 겹치면 충돌로 본다.
- 아래 경우는 모두 충돌이다:
  - 새 예약 시작 시간이 기존 예약 구간 안에 포함되는 경우
  - 새 예약 종료 시간이 기존 예약 구간 안에 포함되는 경우
  - 새 예약이 기존 예약 전체를 감싸는 경우
- 아래 경우는 허용한다:
  - 기존 예약 종료 시간과 새 예약 시작 시간이 정확히 같은 경우
  - 예: 기존 예약 10:00~11:00, 새 예약 11:00~12:00
- 이미 예약된 슬롯 영역은 클릭할 수 없어야 한다.
- 단, UI 차단만으로 충분하지 않으므로 서버에서 반드시 최종 충돌 검사를 해야 한다.
- 가능하면 DB 수준에서도 중복 방지 설계를 검토한다.

메인 예약 현황 화면 요구사항

[화면 구조]
- 메인 화면은 회의실 예약 현황을 보여주는 핵심 화면이다.
- 화면은 캘린더형 타임라인 뷰를 사용한다.
- 가로축은 시간이다.
- 세로축은 회의실이다.
- 시간은 30분 단위 슬롯으로 나눈다.
- 각 슬롯은 클릭 가능한 셀 형태로 보인다.
- 예약이 없는 슬롯은 예약 가능 상태로 표시한다.
- 예약이 있는 슬롯 구간은 예약 블록으로 표시한다.
- 예약 블록은 빈 슬롯과 시각적으로 명확히 구분되어야 한다.

[슬롯 기반 예약 생성 UX]
- 플로팅 액션 버튼은 사용하지 않는다.
- 사용자가 빈 슬롯을 클릭하면 예약 생성 모달이 열려야 한다.
- 모달에는 클릭한 셀의 정보가 기본값으로 들어가야 한다:
  - 회의실
  - 날짜
  - 시작 시간
- 사용자는 종료 시간과 예약 목적을 입력한 뒤 저장할 수 있어야 한다.

[예약 블록 클릭 UX]
- 예약 블록은 모두 클릭 가능해야 한다.
- 남의 예약을 클릭하면 조회 전용 상세 모달이 열린다.
- 내 예약을 클릭하면 상세 모달이 열린다.
- 내 예약 상세 모달에서는 편집 버튼과 취소 버튼이 노출된다.
- 편집 버튼 클릭 시 같은 모달이 editable 상태로 전환된다.
- 남의 예약에서는 편집 버튼과 취소 버튼이 노출되지 않는다.

[예약 가능/불가 처리]
- 이미 예약된 슬롯은 클릭할 수 없어야 한다.
- 종료 시간 선택 단계에서 예약 가능 여부를 즉시 판단해야 한다.
- 현재 예약이 불가능한 시간일 경우 즉시 안내 문구를 노출해야 한다.
- 안내 문구 예시: 현재 예약이 불가능한 시간입니다.

[예약 현황 가시성]
- 사용자는 어떤 회의실이 언제 비어 있는지 빠르게 파악할 수 있어야 한다.
- 사용자는 어떤 회의실이 언제 예약되어 있는지 빠르게 파악할 수 있어야 한다.
- 예약 블록만 보고도 대략적인 사용 길이를 인지할 수 있어야 한다.
- 날짜 이동이 쉬워야 한다.
- 기본 조회 단위는 일간 보기이다.
- 추후 주간 보기로 확장 가능하도록 구조를 설계한다.

필수 화면 목록

1. 로그인 화면
- 회사 이메일 입력
- 비밀번호 입력
- 로그인 버튼
- 로그인 실패 메시지

2. 메인 예약 현황 화면
- 날짜 선택
- 회의실별 30분 단위 타임라인
- 예약 블록 표시
- 빈 슬롯 클릭 시 예약 생성 모달
- 예약 블록 클릭 시 예약 상세 모달
- 내 예약일 경우 편집/취소 가능
- 남의 예약일 경우 조회만 가능
- 예약 가능/불가 상태가 직관적으로 보여야 함

3. 내 정보 화면
- 사용자 기본 정보
- 내 예약 목록 조회
- 비밀번호 변경 기능
- 내가 예약한 목록에는 회의실명, 날짜, 시작 시간, 종료 시간, 예약 목적, 예약 상태를 표시할 수 있어야 한다.

주요 데이터 엔티티

1. User
- id
- company_email (unique, not null)
- name
- password_hash
- is_active
- created_at
- updated_at
- password_changed_at (optional)

2. MeetingRoom
- id
- name
- location (optional)
- description (optional)
- is_active
- created_at
- updated_at

3. Reservation
- id
- user_id
- meeting_room_id
- reservation_date
- start_datetime
- end_datetime
- purpose (optional)
- status
- created_at
- updated_at

4. Session
- id
- user_id
- token_hash
- expires_at
- created_at
- updated_at

데이터 및 정책 세부 규칙

[User]
- company_email은 unique 해야 한다.
- 로그인 아이디는 company_email이다.
- 초기 비밀번호는 모든 사용자에게 동일한 공통값으로 설정한다.
- 실제 저장 시 초기 비밀번호도 해시로 저장해야 한다.

[MeetingRoom]
- 비활성화된 회의실은 예약 대상에서 제외할 수 있어야 한다.
- 회의실 정렬 순서는 이름 또는 관리 기준 순서를 따를 수 있다.

[Reservation]
- status는 최소한 active, cancelled 를 가질 수 있다.
- 취소된 예약은 타임라인에서 기본적으로 비노출 처리한다.
- 예약 수정 시에도 start_datetime, end_datetime 기준으로 충돌 검사를 수행해야 한다.

[Session]
- 로그인 성공 시 서버는 세션을 생성할 수 있어야 한다.
- 세션은 HTTP-only 쿠키 기반으로 관리할 수 있어야 한다.
- 실제 저장 시 세션 토큰 원문이 아니라 해시 값을 저장해야 한다.

예약 상세에서 보여줄 정보
- 회의실명
- 예약자 이름
- 예약 날짜
- 시작 시간
- 종료 시간
- 예약 목적
- 사용 시간은 표시하지 않는다.
- 이메일은 상세 정보에 표시하지 않는다.

비기능 요구사항
- 웹 기반 서비스여야 한다.
- 반응형 웹이어야 한다.
- 데스크탑, 태블릿, 모바일에서 사용 가능해야 한다.
- 서버에서 예약 충돌 검사를 반드시 수행해야 한다.
- UI는 단순하고 직관적이어야 한다.
- 처음 사용하는 사용자도 별도 설명 없이 예약 가능해야 한다.
- 추후 관리자 기능, 회의실 관리 기능, 인증 고도화가 가능하도록 구조를 단순하게 설계해야 한다.

반응형 요구사항

[데스크탑]
- 회의실별 타임라인을 한 화면에서 최대한 많이 보여줄 수 있어야 한다.
- 시간축과 회의실축이 명확해야 한다.
- 예약 블록과 빈 슬롯 구분이 쉬워야 한다.

[태블릿]
- 기본 구조는 유지하되 정보 밀도를 일부 줄일 수 있다.
- 스크롤 UX가 과도하게 불편하지 않아야 한다.

[모바일]
- 타임라인이 너무 좁아질 경우 가로 스크롤을 허용할 수 있다.
- 날짜 선택과 회의실 확인이 가능해야 한다.
- 슬롯 클릭 후 모달에서 예약 입력이 무리 없이 가능해야 한다.
- 정보 과밀로 인해 예약/조회가 어려워지지 않아야 한다.

권장 기술 방향
- 프론트엔드: Next.js
- 언어: TypeScript
- 스타일: Tailwind CSS
- 백엔드: Next.js API Route 또는 Route Handler
- DB: PostgreSQL
- ORM: Prisma
- Auth: 자체 로그인 + HTTP-only 쿠키 세션
- 배포: Vercel

구현 시 중요한 개발 원칙
- 예약 가능 여부를 프론트 UI에서 먼저 표현하되 서버에서 최종 검증해야 한다.
- 타임라인 UI는 조회 성능과 렌더링 단순성을 고려해 설계해야 한다.
- 예약 생성 모달은 클릭한 슬롯의 날짜/시간/회의실을 자동 주입해야 한다.
- 예약 상세 모달은 내 예약 여부에 따라 조회 전용 / 편집 가능 상태를 구분해야 한다.
- 이미 예약된 시간대는 빈 슬롯 클릭 방식으로는 선택할 수 없어야 한다.
- 예약 블록은 모두 클릭 가능해야 한다.
- 모든 시간 계산은 일관된 기준으로 처리해야 한다.
- 추후 관리자 기능 추가를 고려해 User, MeetingRoom, Reservation, Session 구조를 분리해야 한다.

Codex 작업 지침
- 이 요구사항을 임의로 변경하지 말 것
- MVP 범위를 불필요하게 확장하지 말 것
- 먼저 개발 명세, 데이터 모델, 예약 충돌 로직, API, UI 순서로 설계할 것
- 예약 충돌 방지는 프론트가 아니라 서버 중심으로 설계할 것

- Next.js + TypeScript + Tailwind + PostgreSQL + Prisma 조합을 우선 기준으로 삼을 것
