- 날짜별 예약 현황 조회

- 빈 슬롯 클릭 기반 예약 생성 모달
- 예약 상세 조회
- 내 예약 편집
- 내 예약 취소
- 내 정보 화면
- 동일 회의실 시간 중복 예약 방지
- 반응형 웹 지원

### 2.2 MVP 제외 범위
- 회원가입
- 이메일 인증
- SSO / Google Workspace 연동
- 관리자 대시보드
- 반복 예약

- 외부 캘린더 연동
- 회의실 장비 관리

## 3. 제품 목표
- 회사 구성원이 특정 날짜의 회의실 예약 현황을 직관적으로 확인할 수 있어야 한다.
- 사용자는 빈 시간 슬롯을 클릭하여 중복 없이 회의실을 예약할 수 있어야 한다.
- 사용자는 자신의 예약만 수정 및 취소할 수 있어야 한다.
- 반응형 웹 환경에서 데스크탑, 태블릿, 모바일 사용이 가능해야 한다.

## 4. 핵심 사용자와 전제

### 4.1 핵심 사용자
- 회사 내 회의실을 사용하는 일반 구성원

### 4.2 서비스 전제
- 사내 구성원 전용 서비스다.
- 로그인 식별자는 회사 이메일이다.
- 회사 이메일은 회원 DB에서 unique 값이다.
- 회원은 관리자가 사전에 DB에 등록한다.
- 최초 로그인은 회사 이메일 + 공통 초기 비밀번호 방식이다.
- 사용자는 로그인 후 비밀번호를 변경할 수 있다.
- 초기 비밀번호 변경을 강제하거나 별도 안내하지 않는다.

## 5. 사용자 시나리오
1. 사용자는 회사 이메일과 비밀번호로 로그인한다.
2. 사용자는 메인 화면에서 특정 날짜의 회의실 예약 현황을 확인한다.
3. 화면은 회의실별 30분 단위 타임라인으로 표시된다.
4. 사용자는 빈 슬롯을 클릭한다.
5. 선택한 슬롯 정보를 기본값으로 포함한 예약 생성 모달이 열린다.
6. 사용자는 종료 시간과 예약 목적을 입력한다.
7. 서버는 동일 회의실 기준 시간 중복 여부를 최종 검증한다.
8. 중복이 없으면 예약이 생성된다.
9. 생성된 예약은 즉시 타임라인에 반영된다.
10. 사용자는 예약 블록을 클릭하여 상세 내용을 확인한다.
11. 내 예약이면 수정 또는 취소할 수 있다.
12. 남의 예약이면 조회만 가능하다.
13. 사용자는 내 정보 화면에서 내 예약 목록과 비밀번호 변경 기능을 사용한다.

## 7. 기능 명세

### 7.1 인증/회원

#### 로그인
- 사용자는 `company_email`과 비밀번호로 로그인한다.
- 로그인 성공 시 메인 예약 현황 화면으로 이동한다.
- 로그인 실패 시 실패 메시지를 표시한다.

#### 회원 생성 정책
- 회원가입 UI는 제공하지 않는다.
- 회원 레코드는 관리자가 사전 등록한다.
- 모든 회원은 최초 동일한 공통 초기 비밀번호를 가진다.
- 비밀번호는 해시로 저장한다.

#### 비밀번호 변경
- 로그인한 사용자는 내 정보 화면에서 비밀번호를 변경할 수 있다.
- 초기 비밀번호 변경 안내 배너, 첫 로그인 유도 팝업은 구현하지 않는다.

### 7.2 회의실
- 시스템은 여러 개의 회의실을 관리할 수 있어야 한다.
- 회의실 수는 최대 4개 수준을 가정한다.
- 각 회의실은 이름을 필수로 가지며, 위치와 설명은 선택값이다.
- 비활성화된 회의실은 예약 대상에서 제외할 수 있어야 한다.
- 회의실 정렬은 이름 또는 관리 기준 순서를 따를 수 있다.

### 7.3 예약 조회
- 메인 화면은 일간 보기 기준으로 동작한다.
- 사용자는 날짜를 이동하며 해당 날짜의 예약 현황을 확인할 수 있다.
- 시간 범위는 매일 06:00부터 24:00까지다.
- 세로축은 회의실, 가로축은 시간이다.
- 시간은 30분 단위 슬롯으로 표시한다.
- 예약 없는 슬롯은 예약 가능 상태로 표시한다.
- 예약 있는 구간은 예약 블록으로 표시한다.
- 예약 블록은 빈 슬롯과 시각적으로 명확히 구분되어야 한다.
- 취소된 예약은 기본적으로 타임라인에서 노출하지 않는다.

### 7.4 예약 생성
- 사용자는 빈 슬롯을 클릭해 예약 생성 모달을 연다.
- 모달 기본값에는 회의실, 날짜, 시작 시간이 자동 반영된다.
- 사용자는 종료 시간과 예약 목적을 입력한다.
- 예약 목적은 선택 입력이다.
- 시작 시간과 종료 시간은 30분 단위여야 한다.
- 최소 예약 단위는 30분이다.
- 최대 예약 시간 제한은 없다.
- 다른 회의실은 동일 시간대에 동시에 예약 가능하다.
- 같은 사용자가 같은 시간대에 다른 회의실을 중복 예약하는 것도 허용한다.
- 예약 성공 시 타임라인에 즉시 반영한다.

### 7.5 예약 수정
- 내 예약인 경우에만 수정할 수 있다.
- 수정 가능한 항목은 시작 시간, 종료 시간, 예약 목적이다.
- 회의실 변경은 불가하다.
- 수정 UI는 별도 모달이 아니라 동일 상세 모달의 편집 모드 전환 방식이다.
- 수정 시에도 시작 시간과 종료 시간은 30분 단위여야 한다.
- 수정 시에도 서버에서 동일 회의실 기준 중복 여부를 최종 검증한다.
- 수정 가능 시점이 지난 경우 정확한 안내 메시지를 보여준다.

### 7.6 예약 취소
- 내 예약인 경우에만 취소할 수 있다.
- 취소 버튼 클릭 후 확인 모달을 한 번 더 보여준다.
- 취소 확정 시 예약은 즉시 취소된다.
- 취소된 예약은 타임라인에서 즉시 사라져야 한다.
- 예약 취소는 시작 시간과 무관하게 언제든 가능하다.

### 7.7 예약 상세 조회
- 예약 블록은 모두 클릭 가능해야 한다.
- 남의 예약은 조회 전용 상세 모달로 연다.
- 내 예약은 상세 모달에서 편집 버튼과 취소 버튼을 노출한다.
- 남의 예약에는 편집 버튼과 취소 버튼을 노출하지 않는다.

## 8. 예약 정책

### 8.1 시간 정책
- 과거 날짜는 예약 생성이 불가하다.
- 오늘 날짜의 현재 시각 이전 시작 시간은 예약 생성이 불가하다.
- 기존 예약 수정은 예약 시작 전까지만 가능하다.
- 예약 시작 시간이 지난 뒤에는 수정할 수 없다.
- 예약 취소는 시작 시간과 무관하게 언제든 가능하다.

### 8.2 충돌 정책
- 동일 회의실에서 예약 시간 구간이 조금이라도 겹치면 충돌이다.
- 아래 경우는 모두 충돌로 본다.
  - 새 예약 시작 시간이 기존 예약 구간 안에 포함되는 경우
  - 새 예약 종료 시간이 기존 예약 구간 안에 포함되는 경우
  - 새 예약이 기존 예약 전체를 감싸는 경우
- 아래 경우는 허용한다.
  - 기존 예약 종료 시간과 새 예약 시작 시간이 정확히 같은 경우
- UI에서 이미 예약된 슬롯은 클릭할 수 없어야 한다.
- 단, UI 차단만으로 충분하지 않으며 서버에서 반드시 최종 충돌 검사를 수행해야 한다.
- 가능하면 DB 수준에서도 중복 방지 설계를 검토한다.

### 8.3 예약 가능/불가 처리
- 이미 예약된 슬롯은 클릭할 수 없어야 한다.
- 종료 시간 선택 단계에서 예약 가능 여부를 즉시 판단해야 한다.
- 현재 예약이 불가능한 시간일 경우 즉시 안내 문구를 노출해야 한다.
- 안내 문구 예시: `현재 예약이 불가능한 시간입니다.`

## 9. 데이터 모델 명세

### 9.1 User
| 필드명 | 타입 | 제약/설명 |
| --- | --- | --- |
| id | uuid | PK |
| company_email | text | unique, not null, 로그인 아이디 |
| name | text | not null |
| password_hash | text | not null |
| is_active | boolean | not null |
| created_at | timestamptz | not null |
| updated_at | timestamptz | not null |
| password_changed_at | timestamptz | nullable |

### 9.2 MeetingRoom
| 필드명 | 타입 | 제약/설명 |
| --- | --- | --- |
| id | uuid | PK |
| name | text | not null |
| location | text | nullable |
| description | text | nullable |
| sort_order | integer | not null, 기본 정렬 순서 |
| is_active | boolean | not null |
| created_at | timestamptz | not null |
| updated_at | timestamptz | not null |

### 9.3 Reservation
| 필드명 | 타입 | 제약/설명 |
| --- | --- | --- |
| id | uuid | PK |
| user_id | uuid | FK -> User.id, not null |
| meeting_room_id | uuid | FK -> MeetingRoom.id, not null |
| reservation_date | date | not null |
| start_datetime | timestamptz | not null |
| end_datetime | timestamptz | not null |
| purpose | text | nullable |
| status | text | `active`, `cancelled` |
| created_at | timestamptz | not null |
| updated_at | timestamptz | not null |

### 9.4 Session
- 세션 저장소는 DB 테이블이 아니라 서명된 HTTP-only 쿠키를 사용한다.
- 쿠키 payload에는 사용자 식별자, 만료 시각, 자동 로그인 여부, 비밀번호 변경 검증용 값만 담는다.
- 요청 시에는 서명 검증 후 현재 사용자 상태를 DB에서 다시 확인한다.

### 9.5 데이터 규칙
- `company_email`은 unique 여야 한다.
- 초기 비밀번호도 해시로 저장해야 한다.
- 예약 충돌 검사는 `start_datetime`, `end_datetime` 기준으로 수행한다.
- 취소된 예약은 기본 조회에서 제외한다.
- 회의실 정렬은 `sort_order` 우선, 이후 `name` 보조 기준으로 처리한다.
- 모든 시간 저장은 `timestamptz` 기준으로 처리한다.
- 세션은 HTTP-only 쿠키 기반으로 운용하며, 요청마다 현재 사용자 상태를 DB에서 다시 확인한다.
- 예약 중복 방지는 서버 검증 외에 DB exclusion constraint 수준에서도 검토한다.

## 10. API 초안

### 10.1 인증

#### `POST /api/auth/login`
- 목적: 회사 이메일 + 비밀번호 로그인
- 요청값
  - `companyEmail`
  - `password`
- 성공 결과
  - 인증 세션 생성
  - HTTP-only 쿠키 설정
  - 로그인 사용자 기본 정보 반환
- 실패 결과
  - 인증 실패 메시지 반환

#### `POST /api/auth/logout`
- 목적: 현재 로그인 세션 종료
- 제약
  - 로그인 사용자만 호출 가능
- 처리
  - 세션 무효화
  - HTTP-only 쿠키 제거

#### `POST /api/auth/change-password`
- 목적: 로그인 사용자 비밀번호 변경
- 요청값
  - `currentPassword`
  - `newPassword`
- 제약
  - 로그인 사용자만 호출 가능

### 10.2 회의실

#### `GET /api/meeting-rooms`
- 목적: 활성 회의실 목록 조회
- 응답값
  - 회의실 id
  - 이름
  - 위치
  - 설명
  - 정렬 순서에 필요한 값

### 10.3 예약

#### `GET /api/reservations?date=YYYY-MM-DD`
- 목적: 특정 날짜의 예약 현황 조회
- 응답 범위
  - 활성 회의실 기준 예약 목록
  - `status=active` 예약만 포함
  - 예약 블록 렌더링에 필요한 회의실, 예약자, 시간, 목적 정보 포함

#### `POST /api/reservations`
- 목적: 예약 생성
- 요청값
  - `meetingRoomId`
  - `reservationDate`
  - `startDatetime`
  - `endDatetime`
  - `purpose`
- 서버 검증
  - 로그인 사용자 여부
  - 과거 날짜 여부
  - 오늘 기준 현재 시각 이전 시작 시간 여부
  - 30분 단위 여부
  - 종료 시간이 시작 시간 이후인지 여부
  - 동일 회의실 충돌 여부
- 성공 결과
  - 생성된 예약 반환

#### `GET /api/reservations/:id`
- 목적: 예약 상세 조회
- 응답값
  - 회의실명
  - 예약자 이름
  - 예약 날짜
  - 시작 시간
  - 종료 시간
  - 예약 목적
- 비노출 값
  - 예약자 이메일
  - 사용 시간

#### `PATCH /api/reservations/:id`
- 목적: 내 예약 수정
- 요청값
  - `startDatetime`
  - `endDatetime`
  - `purpose`
- 제약
  - 본인 예약만 가능
  - 회의실 변경 불가
  - 예약 시작 전까지만 가능
- 서버 검증
  - 30분 단위 여부
  - 종료 시간이 시작 시간 이후인지 여부
  - 동일 회의실 충돌 여부

#### `POST /api/reservations/:id/cancel`
- 목적: 내 예약 취소
- 제약
  - 본인 예약만 가능
- 처리
  - 상태를 `cancelled`로 변경
  - 응답 후 타임라인에서 즉시 제거 가능해야 함

#### `GET /api/me/reservations`
- 목적: 내 정보 화면의 내 예약 목록 조회
- 응답값
  - 회의실명
  - 날짜
  - 시작 시간
  - 종료 시간
  - 예약 목적
  - 예약 상태

## 11. UI 상세 명세

- 가로축: 06:00~24:00, 30분 단위

  - 이메일
  - 사용 시간

  - 편집 버튼 노출

- 동일 모달 내부에서 전환

  - 종료 시간
  - 예약 목적

#### 예약 현황 가시성
- 사용자는 어떤 회의실이 언제 비어 있는지 빠르게 파악할 수 있어야 한다.
- 사용자는 어떤 회의실이 언제 예약되어 있는지 빠르게 파악할 수 있어야 한다.
- 예약 블록의 길이는 예약 시간 구간에 비례해 표현되어야 하며, 블록만 보고도 대략적인 사용 길이를 인지할 수 있어야 한다.
- 날짜 이동은 쉬워야 한다.
- 타임라인/조회 구조는 현재 MVP에서는 일간 보기만 제공하되, 추후 주간 보기로 확장 가능하도록 설계한다.

### 11.2 로그인 화면
- 입력값 검증 실패 또는 로그인 실패 시 오류 메시지 표시
- 회원가입 링크나 별도 계정 생성 흐름은 제공하지 않는다.

### 11.3 내 정보 화면
- 사용자 기본 정보 표시
- 내 예약 목록 표시
- 비밀번호 변경 폼 제공
- 내 예약 목록에는 회의실명, 날짜, 시작 시간, 종료 시간, 예약 목적, 예약 상태를 표시해야 한다.

## 12. 반응형 명세

### 12.1 데스크탑
- 회의실별 타임라인을 한 화면에서 최대한 많이 보여줄 수 있어야 한다.
- 시간축과 회의실축이 명확해야 한다.
- 예약 블록과 빈 슬롯 구분이 쉬워야 한다.

### 12.2 태블릿
- 기본 구조는 유지하되 정보 밀도를 일부 줄일 수 있다.
- 스크롤 UX가 과도하게 불편하지 않아야 한다.

### 12.3 모바일
- 타임라인이 좁아질 경우 가로 스크롤을 허용할 수 있다.
- 날짜 선택과 회의실 확인이 가능해야 한다.
- 슬롯 클릭 후 모달에서 예약 입력이 가능해야 한다.
- 정보 과밀로 인해 예약/조회가 어려워지지 않아야 한다.

## 13. 상태 및 권한 규칙

### 13.1 사용자 권한
- 로그인 사용자는 예약 현황을 조회할 수 있다.
- 로그인 사용자는 빈 슬롯에 대해 예약 생성이 가능하다.
- 로그인 사용자는 자신의 예약만 수정/취소할 수 있다.
- 로그인 사용자는 다른 사용자의 예약을 조회만 할 수 있다.

### 13.2 예약 상태
- `active`: 현재 유효한 예약
- `cancelled`: 취소된 예약

## 14. 구현 원칙
- 예약 가능 여부는 프론트에서 먼저 표현하되 서버에서 최종 검증한다.
- 예약 충돌 방지는 프론트가 아니라 서버 중심으로 설계한다.
- 타임라인 UI는 조회 성능과 렌더링 단순성을 고려해 설계한다.
- 모든 시간 계산은 일관된 기준으로 처리한다.
- 날짜별 일간 조회를 우선 구현하되, 조회 모델과 타임라인 렌더링 구조는 추후 주간 보기 확장이 가능하도록 설계한다.
- 추후 관리자 기능, 회의실 관리 기능, 인증 고도화가 가능하도록 `User`, `MeetingRoom`, `Reservation` 구조를 분리한다.
- 추후 관리자 기능, 회의실 관리 기능, 인증 고도화가 가능하도록 `User`, `MeetingRoom`, `Reservation` 구조를 분리한다.
