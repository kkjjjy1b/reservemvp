# Important Change Log

PM, Reviewer, Builder가 배포 전에 확인해야 할 중요한 변경만 남긴 요약 로그다.

## 기록 원칙
- UI 미세 수정, 문구 수정, 오타 수정은 기록하지 않는다.
- 요구사항 해석, DB 구조, 인증, 권한, API 계약, 핵심 UI 흐름 변경만 남긴다.
- 같은 주제의 반복 수정은 최종 결정만 남기고 압축한다.

## 2026-03-13
- 변경: 사용자 표시 이름을 화면에서 가공하지 않고 DB 원본(`users.name`) 자체를 최종 표시형으로 맞췄다. 현재 기본 데이터는 `김민지`, `박준호`, `이서연`, `최도윤`이다.
- 이유: 이름 표시 규칙을 UI 단계에서 뒤집으면 예외가 늘고, 화면마다 일관성이 깨질 수 있다.
- 영향: 사용자 이름을 쓰는 타임라인, 예약 상세, 내 정보 화면 전반
- 확인 필요: 운영에서도 표시 이름을 공백 없는 실명 기준으로 유지할지 확인

## 2026-03-13
- 변경: 회의실 수용 인원을 랜덤 표시에서 DB 필드(`meeting_rooms.capacity`)로 전환하고, 기본 회의실 데이터를 `와이낫(20명)`, `두잇(10명)`, `쏘왓(5명)`으로 정리했다.
- 이유: 실제 운영 데이터 기준으로 표시해야 하고, 랜덤값은 검수와 운영 모두에 혼란을 준다.
- 영향: Prisma schema, migration, seed, 타임라인 회의실 헤더
- 확인 필요: 운영 초기 데이터도 동일한 회의실명/인원 기준으로 갈지 확인

## 2026-03-12
- 변경: 메인 화면을 `시간 세로축 + 회의실 컬럼` 기반의 일간 타임라인으로 전면 교체했다. 빈 슬롯 클릭으로 예약 생성, 예약 카드 클릭으로 상세/수정/취소까지 이어지는 흐름을 완성했다.
- 이유: MVP 핵심 기능이 회의실 현황 조회와 빠른 예약 생성이기 때문이다.
- 영향: 메인 페이지 UI, 예약 생성 모달, 예약 상세 모달, 예약 API
- 확인 필요: 현재 타임라인 인터랙션과 모달 흐름이 MVP 최종 UX로 충분한지 확인

## 2026-03-12
- 변경: 예약마다 고정 포인트 컬러를 저장하도록 `reservations.color_key`를 추가했다.
- 이유: 예약 블록을 시각적으로 빠르게 구분해야 하고, 새로고침 후에도 같은 색을 유지해야 한다.
- 영향: DB migration, 예약 생성 API, 타임라인 카드 색상
- 확인 필요: 현재 5종 색상 팔레트를 그대로 유지할지 확인

## 2026-03-12
- 변경: 인증 방식을 `자체 로그인 + HTTP-only 쿠키 세션`으로 확정했다. 로그인, 로그아웃, 자동 로그인, 내 정보 조회, 비밀번호 변경 API를 구현했다.
- 이유: 외부 인증 서비스 없이도 MVP 요구사항을 가장 단순하게 만족할 수 있다.
- 영향: 인증 API, 세션 테이블, 로그인 화면, 내 정보 화면
- 확인 필요: 자동 로그인 유지 기간과 초기 비밀번호 운영 정책 확인

## 2026-03-12
- 변경: `/me` 화면을 추가해 사용자 기본 정보, 프로필 수정, 비밀번호 변경, 내 예약 목록 조회를 한 화면에 구성했다.
- 이유: 사용자가 자기 정보와 내 예약을 별도 화면 이동 없이 관리할 수 있어야 한다.
- 영향: 내 정보 화면 UI, `PATCH /api/me/profile`, 내 예약 목록 API
- 확인 필요: 현재 `/me` 정보 구조와 아이콘 액션 구성이 최종 운영 화면으로 적절한지 확인

## 2026-03-12
- 변경: PostgreSQL 수준의 예약 충돌 방지를 위해 exclusion constraint 기반 구조를 도입했다.
- 이유: 앱 로직만으로는 경쟁 상태를 완전히 막을 수 없기 때문이다.
- 영향: Prisma migration 이후 후속 SQL migration, 예약 생성/수정 정책
- 확인 필요: 운영 DB에도 동일 migration 순서로 적용할 것

## 2026-03-12
- 변경: 실행 스택을 `Next.js + TypeScript + Tailwind + Prisma + PostgreSQL`로 고정하고, 로컬/검수 실행 스크립트를 정리했다.
- 이유: 배포 전까지는 기술 선택을 더 흔들지 않고 실행 가능성과 재현 가능성을 우선해야 한다.
- 영향: package scripts, README 실행 가이드, review 서버 운영 방식
- 확인 필요: 배포 플랫폼 선택 후 운영 실행 방식과 맞는지 확인
